# Статусы и развёртывание

После отправки заказ проходит несколько этапов: оформление изменения в Git (merge request), его слияние и развёртывание в кластере. Текущий этап виден в списке заказов и на странице заказа.

## Жизненный цикл заказа

| Статус | Что означает |
|--------|--------------|
| Черновик | Заказ создан, но ещё не отправлен |
| MR открыт | По заказу создан запрос на слияние (MR) в Git, ждёт слияния |
| MR влит | Изменение влито в Git, начинается развёртывание |
| Разворачивается | Платформа приводит кластер к заказанному состоянию |
| Готов | Сервис развёрнут и работает (Healthy) |
| Проблемы | Сервис развёрнут, но в нездоровом состоянии (Degraded) |

Есть также статусы для удаления сервиса (запрошено удаление, удаление влито, удалён). Заказ применяется только после слияния MR - до этого он остаётся в статусе «MR открыт».

## Где смотреть развёртывание

Состояние развёртывания платформа получает из системы доставки (GitOps). На странице заказа отображается итоговое состояние сервиса: синхронизирован ли он с заказанной конфигурацией и здоров ли.

## Действия на странице заказа

В зависимости от статуса и ваших прав на странице заказа доступны действия:

- **Изменить** - поменять параметры черновика или открыть изменение для уже работающего сервиса;
- **Обновить** - перейти на новую версию сервиса;
- **Синхронизировать** - попросить систему доставки заново привести кластер к заказанному состоянию;
- **Удалить** - убрать сервис (оформляется как отдельное изменение в Git).

## История действий

На странице заказа есть хронология: кто и когда создал заказ, отправил, какие были переходы статусов, открытые запросы на слияние (merge requests) и ссылки на них. Это полный аудит-след по сервису - удобно при разборе инцидентов и при передаче дел.

## Дрейф (расхождение с Git)

Дрейф - это когда конфигурация сервиса в Git отличается от того, что заказано через портал (например, кто-то поправил файлы вручную). Портал замечает расхождение и показывает баннер на заказе.

Что можно сделать:

- **Подтянуть из Git** - принять то, что сейчас в Git, как актуальное состояние заказа;
- **отредактировать заказ** - наоборот, вернуть конфигурацию к заказанной через портал.

> [!TIP]
> Дрейф - это сигнал, а не ошибка. Он помогает поймать изменения «в обход портала» раньше, чем они приведут к проблемам.

## Импорт заказов, созданных вне портала

Портал постоянно следит за GitOps-репозиторием. Если сервис завели напрямую в Git, минуя портал, он не теряется: портал заметит его и подхватит как обычный заказ. Такой заказ помечается значком **Импорт** в списке - сразу видно, что он появился извне.

Портал подхватывает не любой сервис из Git, а только тот, что оформлен по правилам платформы - так же, как это сделал бы сам портал при заказе. Каждый сервис в Git описан папкой с двумя файлами: какой чарт и версию ставить и в какой кластер (файл приложения, `application.yaml`) и какими параметрами его настроить (файл значений, `values.yaml`). Как именно разложены папки и файлы - в разделе [Структура сервисов в Git](git-structure). Чтобы сервис был импортирован, должны выполняться три условия:

- **Папка сервиса заполнена полностью** - в ней есть оба файла: и `application.yaml`, и `values.yaml` с параметрами. Если папка неполная (например, нет `values.yaml`), портал не сможет показать и редактировать такой заказ, поэтому пропускает его.
- **Сервис собран по стандартной схеме платформы** - `application.yaml` выглядит ровно так, как его сформировал бы портал для такого заказа (тот же чарт, та же структура). Это отсекает «чужие» сервисы, развёрнутые в том же Git по другим правилам: портал не берёт под управление то, что создавалось не по его конвенции.
- **Параметры корректны** - значения в `values.yaml` проходят проверку по форме заказа этого сервиса. Иначе заказ нельзя было бы открыть и изменить через портал.

Если хотя бы одно условие не выполнено, сервис не импортируется и остаётся жить в Git сам по себе. После успешного импорта заказом можно управлять как обычно: смотреть статус, редактировать, обновлять, удалять.

> [!NOTE]
> Импорт - это разовое подхватывание целого нового сервиса из Git. Не путайте с дрейфом: дрейф отслеживает изменения уже известного портала заказа, а импорт заводит в портал то, чего он раньше не знал.

## Если что-то пошло не так

- **Заказ долго не разворачивается** - скорее всего, не влит merge request: откройте его по ссылке на странице заказа и попросите смержить того, у кого есть права на репозиторий команды.
- **Развёртывание не доходит до «Готов»** - откройте страницу заказа: там отображается причина (например, ошибка проверки параметров).
- **Нужны изменения** - не создавайте новый заказ, а отредактируйте существующий.

> [!TIP]
> Параметры сервиса можно менять и напрямую в Git, но через портал обычно удобнее. Прямые правки в Git портал не подхватывает автоматически: он отметит расхождение (дрейф), которое потом нужно будет свести вручную (см. «Дрейф» выше).
